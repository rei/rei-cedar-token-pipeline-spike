# File Dictionary

This document is a governance map of repository files and file groups.
It answers three questions for each entry:

1. What is this file for?
2. What does it contain?
3. How should changes be governed?

## Governance Tiers

- Tier 1: Contract files. Breaking changes require ADR or explicit approval in PR.
- Tier 2: Build logic files. Changes require tests or validation output.
- Tier 3: Input/output artifacts. Changes require source traceability.
- Tier 4: Local/editor/tooling. Changes are optional and should be isolated.

## Recommended Governance Workflow

1. Classify each changed file by tier in PR description.
2. Enforce required review rules by tier:
   - Tier 1: Design + engineering approval.
   - Tier 2: Engineering approval + tests.
   - Tier 3: Traceability to source or build output.
   - Tier 4: Optional, separate commit preferred.
3. Require a short impact note for Tier 1 and Tier 2 changes.
4. Keep generated artifacts and logic changes in separate commits when possible.

## Maintenance Rules

- Update this dictionary whenever a new top-level folder or pipeline stage is added.
- Prefer path patterns for generated families and concrete paths for contracts.
- Keep owner groups aligned with `.github/CODEOWNERS` path patterns.
