# ADR‚Äë0013: Consumer Token Context Surface

## Status

Draft

## Context

External consumers of this package ‚Äî design engineers, design tooling integrations, and their AI agents ‚Äî receive token artifacts (CSS custom properties, JSON, iOS color assets) with no architectural context attached. From a published artifact alone, a consumer cannot determine:

- Why a specific token resolves the way it does
- Which decisions govern the token structure they are consuming
- What is safe to extend vs. what is a governed invariant they must not override
- Whether a token is stable, experimental, or deprecated

This is a distinct hallucination vector from internal contributor grounding (ADR‚Äë0012). Even if internal agents are well-grounded, consumer agents that query "how do I use this token system?" have nothing to anchor to and will invent answers from pattern-matching the output artifacts alone.

The current `$extensions.cedar.governance` metadata in `canonical/tokens.json` carries per-token status and usage notes (ADR‚Äë0010), but it does not reference the ADRs that govern the token's structure or resolution behavior. The full ADR catalog is an internal artifact and is not published with the package.

---

## Decision

### 1. Add `adrRef` to governed token governance metadata

Tokens with `$extensions.cedar.governance` entries should include an `adrRef` field pointing to the primary ADR governing their structure or resolution behavior (e.g., `"adrRef": "ADR-0001"`). This is opt-in initially and required for any token whose governance entry is created or updated after this ADR is accepted.

### 2. Publish a machine-readable architecture summary alongside token outputs

A lightweight `agent-context.json` artifact is generated into `dist/` during `pnpm tokens:build`. It contains:

- Token tier model description (options ‚Üí alias ‚Üí component)
- Active invariants ‚Äî `decision` and `consequences` bodies from all `Accepted` and `Implemented` ADRs
- Explicitly excluded patterns ‚Äî extracted `Non-Responsibilities` section bodies from active ADRs
- Package version and ADR catalog version

This is a distilled consumer-facing surface, not a copy of the full catalog.

### 3. Generation is part of the build pipeline

`dist/agent-context.json` is generated by a new build step `docs:agent-context` that reads the ADR catalog, filters to `status: Accepted | Implemented`, and serializes the distilled output. It is never manually edited. The build step fails if the catalog is missing or stale.

### 4. Scope is limited to architectural constraints, not usage documentation

`agent-context.json` does not replace Storybook, Figma documentation, or per-token usage notes. Its sole purpose is to give consumer agents a grounded, structured answer to "what does this system enforce and why?" ‚Äî not "how do I pick a token?"

### 5. Exclude `agent-context.json` from token resolution

Consumers must not treat `dist/agent-context.json` as a token file. The build output structure must make this unambiguous (e.g., separate subdirectory or explicit exclusion pattern in published package exports).

---

## Consequences

**Positive:**
- Consumer agents have a structured grounding surface without requiring access to the internal ADR directory
- Drift between catalog and published artifact is structurally prevented: generation is sourced from the catalog
- `adrRef` on tokens creates a traceable link from any token to the decision that governs it

**Negative:**
- Adds a build step and a new published artifact to maintain and version
- `adrRef` annotations on tokens require authors to update governance metadata when ADR status changes ‚Äî ongoing maintenance cost
- The distilled surface is only as good as ADR `Decision` and `Consequences` section coverage; poorly-written ADRs produce poor consumer context

**Risks:**
- If `agent-context.json` is not regenerated after an ADR is promoted from `Draft` to `Accepted`, consumer agents receive an incomplete constraint set ‚Äî the pipeline dependency mitigates this but authors must still promote ADR status deliberately
- Publishing architectural intent as a machine-readable artifact increases surface area for competitors to understand the system's governed design decisions
