## Status

Draft

## Context

AI coding agents (GitHub Copilot, autonomous PR agents, and contributor-facing chat) are increasingly used to propose, review, and generate changes in this repository. Without structured grounding, these agents rely on markdown scanning, heuristic pattern-matching, and general training knowledge to interpret the codebase. This produces three categories of failure:

1. **Hallucinated structure** â€” agents propose token shapes, normalization paths, or status values that contradict established ADRs because they inferred intent from code patterns rather than documented decisions.
2. **Architectural drift** â€” agents make locally-correct changes that violate invariants defined in ADRs they were unaware of (e.g., adding a normalization heuristic that ADRâ€‘0002 explicitly prohibits).
3. **Stale context** â€” agents rely on outdated architectural assumptions when ADRs have been superseded or amended.

The `adr-catalog.json` artifact provides a machine-readable index of all ADRs with extracted status, decision, context, and relationship metadata. However, it is not yet wired into agent entry points, so agents currently discover architecture through directory listing rather than structured lookup.

The `.github/copilot-instructions.md` file governs contribution scope and safety rules but does not direct agents to the catalog or describe how to query it.

---

## Decision

### 1. Make the ADR catalog the canonical grounding source for all agents

`architecture/adr-catalog.json` is the governed, always-current machine-readable index of architectural decisions. All agent instructions must reference it as the first lookup step before proposing changes to `src/**`, `architecture/**`, `style-dictionary/**`, or `scripts/**`.

### 2. Extend `copilot-instructions.md` with an explicit agent grounding section

The instructions file must specify:

- `architecture/adr-catalog.json` is the agent entry point for architectural context
- Filter by `status: Accepted | Implemented` for currently enforced constraints
- Filter by `status: Superseded` to identify deprecated approaches before proposing similar patterns
- Agents must cite the governing ADR number in any PR or commit that touches governed behavior
- `Proposed` and `Draft` ADRs are informational â€” not yet binding
- `Accepted` and `Implemented` ADRs are hard constraints that require an ADR amendment to override

### 3. Annotate contract tests with `@enforces` references

Contract tests in `src/contract/` must include a `// @enforces ADR-XXXX` comment linking each test suite to the ADR it validates. This makes the relationship between code enforcement and documented intent explicit and queryable by agents reading the source.

### 4. Add a `constraints` field to the catalog parser

The catalog parser should extract a `constraints` array from ADRs that contain a `## Governed Invariants`, `## Validation Requirements`, or `## Non-Responsibilities` section. This allows agents to query what is explicitly prohibited without reading full ADR prose.

### 5. The catalog is a generated artifact â€” agents must never edit it directly

`architecture/adr-catalog.json` is always regenerated by `pnpm docs:adr:catalog` or the pre-commit hook. Direct edits will be overwritten on next commit.

---

## Consequences

**Positive:**
- Agents have a deterministic, structured path to relevant decisions without requiring full markdown comprehension
- Hallucination of ADR structure, status values, and invariants is reduced: the catalog is always current and contract-enforced
- `@enforces` annotations make coverage gaps visible â€” decisions without a corresponding test are identifiable

**Negative:**
- Agents that cannot read files (prompt-only) cannot benefit from the catalog directly
- The `constraints` extraction adds parser complexity and depends on consistent section naming in ADR authors' writing
- Authors must use governed section headings (`## Governed Invariants`, etc.) for constraints to be captured automatically

**Risks:**
- If ADR coverage is incomplete (non-standard structure), agents may see partial constraint sets and treat silence as permission
- The grounding strategy only applies to agents that receive the instructions file; external forks or CI-only runs do not inherit this context
